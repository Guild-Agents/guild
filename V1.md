description: Enriquece CLAUDE.md explorando el proyecto y especializa los agentes al stack real

---

**Lo que hace:**

```
1. Lee PROJECT.md y CLAUDE.md base
2. Explora estructura del proyecto:
   - package.json / pom.xml / build.gradle (dependencias reales)
   - Carpetas src/, app/, lib/ (arquitectura real)
   - Archivos de configuración (eslint, tsconfig, env.example)
   - Migraciones existentes si las hay
   - README si existe
3. Genera CLAUDE.md enriquecido con:
   - Stack con versiones exactas
   - Estructura de carpetas explicada
   - Convenciones de código detectadas
   - Patrones de arquitectura identificados
   - Variables de entorno conocidas
   - Limitaciones y deuda técnica visible
4. Actualiza agents/*.md añadiendo contexto específico del proyecto
5. Confirma: "Guild v1 listo. Ejecuta /status para ver el estado."
```

## Definición de los 8 Agentes

Cada agente en `.claude/agents/` define **quién es**, no **qué hacer**.

### `advisor.md`
```markdown
---
name: advisor
description: Evalúa ideas y da dirección estratégica antes de comprometer trabajo
---

Eres el Advisor de [PROYECTO]. Tu rol es evaluar ideas desde una perspectiva
estratégica y técnica antes de que el equipo invierta tiempo.

Cuando evalúas una idea:
- Preguntas primero si no tienes suficiente contexto
- Identificas riesgos técnicos y de negocio
- Das una recomendación clara: go/no-go/modificar
- Si es go, defines el scope mínimo viable
- Siempre lees CLAUDE.md antes de evaluar

Tu output es: evaluación + recomendación + scope sugerido.
```

### `product-owner.md`
```markdown
Convierte ideas aprobadas en tareas concretas e implementables.
Produce: título, descripción, criterios de aceptación, tareas técnicas, estimación.
```

### `tech-lead.md`
```markdown
Define el approach técnico. Revisa arquitectura, elige patterns,
identifica riesgos de implementación, define interfaces entre componentes.
```

### `developer.md`
```markdown
Implementa features siguiendo las convenciones del proyecto.
Lee CLAUDE.md antes de tocar código. TDD cuando aplica.
Hace commits atómicos con mensajes descriptivos.
```

### `code-reviewer.md`
```markdown
Revisa código con criterio técnico. Busca: bugs potenciales, violaciones
de patterns del proyecto, deuda técnica, casos no manejados.
Output: lista de issues por severidad (blocker/warning/suggestion).
```

### `qa.md`
```markdown
Diseña y ejecuta casos de prueba. Foco en edge cases, regression,
y criterios de aceptación del PO. Reporta bugs con reproducción exacta.
```

### `bugfix.md`
```markdown
Diagnostica y resuelve bugs. Proceso: reproduce → entiende causa raíz
→ propone fix mínimo → implementa → verifica que no rompe nada más.
```

### `db-migration.md`
```markdown
Maneja cambios de schema. Siempre: migración up + down, verifica
datos existentes, considera performance en producción.
```

## Skills — Los Workflows

### `build-feature/SKILL.md` — Pipeline principal

```yaml
---
name: build-feature
description: Pipeline completo para implementar una feature: evaluación → spec → implementación → review → QA
disable-model-invocation: true
allowed-tools: Task, Read, Bash
---

# build-feature

Ejecuta el pipeline completo de desarrollo para una feature.

## Uso
/build-feature [descripción de la feature]

## Pipeline

### Fase 1 — Evaluación (Advisor)
Usa el Task tool para invocar al Advisor:
- Lee `.claude/agents/advisor.md`
- Evalúa la feature propuesta
- Output: go/no-go + scope

Si no-go: presenta el resultado y detén el pipeline.

### Fase 2 — Especificación (Product Owner)
Usa el Task tool para invocar al Product Owner:
- Lee `.claude/agents/product-owner.md`
- Convierte el scope aprobado en spec técnica
- Output: criterios de aceptación + tareas

### Fase 3 — Approach técnico (Tech Lead)
Usa el Task tool para invocar al Tech Lead:
- Lee `.claude/agents/tech-lead.md`
- Define approach, patterns a usar, archivos a modificar
- Output: plan de implementación

### Fase 4 — Implementación (Developer)
Usa el Task tool para invocar al Developer:
- Lee `.claude/agents/developer.md`
- Implementa según el plan del Tech Lead
- Output: código commiteado

### Fase 5 — Review (Code Reviewer)
Usa el Task tool para invocar al Code Reviewer:
- Lee `.claude/agents/code-reviewer.md`
- Revisa el código implementado
- Si hay blockers: vuelve a Fase 4

### Fase 6 — QA
Usa el Task tool para invocar al QA:
- Lee `.claude/agents/qa.md`
- Ejecuta criterios de aceptación del PO
- Si hay bugs: invoca bugfix.md y repite review

Pipeline completo → actualiza SESSION.md con el resumen.
```

### `council/SKILL.md` — Party Mode

```yaml
---
name: council
description: Convoca múltiples agentes para debatir una decisión importante
disable-model-invocation: true
allowed-tools: Task, Read
---

# council

Cuando necesitas múltiples perspectivas antes de decidir algo importante.

## Uso
/council [pregunta o decisión]

## Consejos disponibles

### council architecture
Invoca en paralelo: Tech Lead + Advisor + Developer
Cada uno da su perspectiva desde su rol.
Luego sintetiza las perspectivas y presenta el debate.

### council feature-scope  
Invoca: Advisor + Product Owner + Tech Lead
Para decidir qué incluir y qué dejar fuera de una feature.

### council tech-debt
Invoca: Tech Lead + Developer + Code Reviewer
Para priorizar qué deuda técnica atender primero.
```

## Posicionamiento en README (borrador)

> **Guild** es un framework de ejecución para Claude Code.  
> No es metodología agile — es estructura sin ceremonia.  
> Implementa la feature esta tarde, no el sprint que viene.
>
> Ideal si: ya usas Claude Code y quieres roles definidos,
> pipelines reales, y contexto que persiste entre sesiones.
>
> No es para ti si: necesitas PRDs formales, planificación de equipo,
> o quieres ser agnóstico al AI tool. Para eso, mira BMAD.
