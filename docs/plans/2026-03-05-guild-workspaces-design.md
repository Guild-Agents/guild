# Guild Workspaces — Design

**Date:** 2026-03-05
**Status:** Approved
**Scope:** v1.2.0 (context + read), v1.2.1 (cross-repo execution)

## Problem

Guild operates on a single project. Real products often span multiple repos (e.g., backend + frontend). Today there is no way to share agents, skills, or context between related repos, and no way for an agent working in one repo to understand or read the other.

## Solution: Workspace in Parent Directory

A workspace groups independent repos under a parent directory with a `guild-workspace.json` manifest and shared config in `.guild/`.

### Structure

```
~/projects/my-product/
  guild-workspace.json              # workspace manifest
  .guild/                           # shared config (NOT .claude/)
    agents/                         # shared agent definitions
    skills/                         # shared skill definitions
  backend/                          # independent git repo
    CLAUDE.md
    .claude/agents/                 # local agents (can override shared)
    .claude/skills/
  frontend/                         # independent git repo
    CLAUDE.md
    .claude/agents/
    .claude/skills/
```

`.guild/` is used at the workspace root to avoid confusion with `.claude/` which belongs to individual repos.

### `guild-workspace.json`

```json
{
  "name": "my-product",
  "members": [
    { "name": "backend", "path": "./backend" },
    { "name": "frontend", "path": "./frontend" }
  ],
  "shared": {
    "agents": ".guild/agents",
    "skills": ".guild/skills"
  }
}
```

Minimal and declarative. No magic.

## Resolution: Merge + Local-Wins

When Guild loads agents/skills for a member repo:

1. Read shared from workspace (`.guild/agents/`)
2. Read local from repo (`.claude/agents/`)
3. Merge: same filename = local wins. Workspace-only files = inherited. Local-only files = kept.

File-level override only — no content-level merge within a file.

## Cross-Repo Context (v1.2.0)

When an agent works in a member repo, CLAUDE.md includes a workspace context section:

```markdown
## Workspace context
- **Workspace:** my-product
- **Members:** backend (this), frontend
- **Frontend stack:** React + Vite (from frontend/PROJECT.md)
- **Frontend structure:** src/components/, src/api/ (from frontend/CLAUDE.md)
```

Generated by reading `PROJECT.md` and `CLAUDE.md` from other member repos. Read-only — agents can read files from other repos but not modify them in v1.2.0.

## CLI Commands

| Command                   | Description                                              |
| ------------------------- | -------------------------------------------------------- |
| `guild workspace init`    | Create `guild-workspace.json`, prompt for members        |
| `guild workspace add <p>` | Add a member repo to the workspace                       |
| `guild workspace status`  | Show workspace, members, and state of each               |

## Code Impact

- **`resolveProjectRoot()`** — extend to also search for `guild-workspace.json` upward and resolve workspace root
- **`files.js`** — new `resolveWorkspace()` function to load and merge config
- **`generators.js`** — `generateClaudeMd()` optionally injects "Workspace context" section
- **`guild init`** — detect if inside a workspace and inform user
- **New**: `src/commands/workspace.js` with subcommands
- **New**: `src/utils/workspace.js` for workspace resolution and merge logic

## Phases

### v1.2.0 — Context + Read

- `guild-workspace.json` manifest
- `.guild/` shared directory with agents and skills
- Merge + local-wins resolution
- Workspace context injection in CLAUDE.md
- Agents can read files from other member repos
- CLI: `guild workspace init`, `guild workspace add`, `guild workspace status`

### v1.2.1 — Cross-Repo Council

When `/council` runs inside a workspace, it automatically enriches each agent's prompt with sibling repo context. No new command or flag needed.

#### What each agent receives

1. **Injected summary** — stack, structure, and current task from each sibling's CLAUDE.md, PROJECT.md, SESSION.md
2. **Absolute paths** — so the agent can read any file from sibling repos for deeper analysis

#### Code changes

- **`src/utils/workspace.js`** — new `collectMemberContext(workspace, currentMember)` function that reads CLAUDE.md/PROJECT.md/SESSION.md from each sibling and returns a structured summary
- **`src/templates/skills/council/SKILL.md`** — Step 2 modified to: detect workspace, call `collectMemberContext`, inject context into each agent's prompt

#### Injected context format

```markdown
## Workspace: my-product

### frontend (sibling — /abs/path/to/frontend)
- **Stack:** React, Vite, TypeScript
- **Structure:** src/components/, src/api/, src/hooks/
- **Current task:** migrating to React 19
You can read any file under /abs/path/to/frontend/ for deeper analysis.
```

#### What does NOT change

- Council types (architecture, feature-scope, tech-debt) — workspace is a layer on top
- Synthesis and options presentation (Steps 3-5)
- Spec document format
- Agents integrate cross-repo impact in their normal arguments (no separate section)

#### Testing

- Unit tests for `collectMemberContext()` — with/without siblings, with/without missing files
- Skill template changes are markdown, validated manually

### Future: v1.2.2+ — Cross-Repo Commands

- Run tests/lint in sibling repos from a skill
- Skills that coordinate execution across repos

## Out of Scope

- Automatic config sync between repos
- Remote workspaces (repos on different machines)
- Monorepo support (single repo with multiple packages)
- Workspace-level git operations
